<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Telegram guide UI → auth status refresh provider | User-triggered action initiates network-backed state refresh |
| Auth refresh result → UI feedback/navigation | Remote auth state controls snackbar content and pop behavior |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | S (Spoofing) | Status check trigger | mitigate | Keep refresh call behind existing authenticated provider path (`authStatusRefreshProvider`); no alternate unauthenticated path introduced. |
| T-02-02 | T (Tampering) | UI state transitions | mitigate | Enforce single in-flight state (`_isCheckingStatus`) with disabled action and deterministic state reset in `finally`. |
| T-02-03 | R (Repudiation) | User feedback events | accept | No audit requirement for this local UX phase; snackbars provide immediate user-visible operation result. |
| T-02-04 | I (Information Disclosure) | Failure feedback text | mitigate | Keep generic error messaging; do not surface token/session internals in snackbar copy. |
| T-02-05 | D (Denial of Service) | Rapid repeated taps | mitigate | Disable Step 3 action during in-flight check and verify behavior through widget tests. |
| T-02-06 | E (Elevation of Privilege) | Success navigation path | mitigate | Only execute success return flow when refreshed auth state is explicitly linked (`is_guest=false`). |
</threat_model>
