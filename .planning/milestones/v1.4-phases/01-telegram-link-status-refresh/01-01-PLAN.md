<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Telegram guide UI → auth provider | User-triggered action causes network-backed auth refresh path |
| auth provider → `/auth/device/` API | Remote response controls `is_guest`-driven navigation/feedback |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | S (Spoofing) | Step 3 refresh trigger | mitigate | Route refresh exclusively through existing authenticated `AuthRepository` flow; do not add alternate unauthenticated client path in UI code. |
| T-01-02 | T (Tampering) | app_version payload field | mitigate | Source app version from `AppConstants.appVersion` used by Settings (D-05), removing stale hardcoded auth-only value. |
| T-01-03 | R (Repudiation) | User-initiated status checks | accept | No audit log requirement in phase scope; user-visible snackbar outcomes provide immediate action trace only. |
| T-01-04 | I (Information Disclosure) | Snackbar/UI feedback | mitigate | Use generic success/failure feedback only; do not expose token/device identifiers in UI messages or errors. |
| T-01-05 | D (Denial of Service) | Repeated Step 3 taps | mitigate | Disable Step 3 action and show in-button spinner during in-flight refresh (D-02) to suppress duplicate requests. |
| T-01-06 | E (Elevation of Privilege) | Navigation on linked status | mitigate | Navigate back only when refreshed state explicitly reports `is_guest=false` from authenticated `/auth/device/` response (D-03). |
</threat_model>
