<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user text/clipboard → submit handler | Telegram ID input from keyboard/clipboard is untrusted data crossing into provider call. |
| app → external Telegram deep link | Opening external bot URL crosses app boundary and may fail/device-route unexpectedly. |
| UI submit → API-backed provider | Untrusted network outcomes cross into user-visible messaging/navigation flow. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-03 | T | telegram_link_guide_screen.dart input path | mitigate | Submit only through existing provider validation and keep explicit invalid-ID messaging; do not bypass provider guards. |
| T-12-04 | D | link submit action | mitigate | Disable/guard Link control while `isSubmitting` to avoid UI-level request storms; rely on provider in-flight suppression. |
| T-12-05 | I | outcome feedback surface | mitigate | Show only outcome-type user messages; never expose raw backend/auth/token details in snackbars. |
</threat_model>
